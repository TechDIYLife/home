使用 Hugging Face Zephyr 和 LangChain 快速构建 RAG

作者：Maria Khalusova

这个教程将展示你如何利用 HuggingFaceH4/zephyr-7b-beta 模型和 LangChain 技术，快速创建一个针对项目 GitHub 问题的 RAG（检索增强生成）。

什么是 RAG?

RAG 是一个广为人知的解决方案，它帮助强大的大语言模型 (LLM) 克服不了解特定内容的局限，这些内容可能是因为它们没有包含在模型的训练数据中，或者即使包含了也容易出现错误。这些内容可能是专有的、敏感的，或者像这个案例中的，频繁更新的内容。

如果你的数据是固定不变的，你或许可以通过对模型进行微调来处理。但在很多情况下，微调不仅成本高昂，而且还可能因为需要应对数据的持续变化而导致“模型偏移”，即模型的行为以不可预见的方式发生改变。

RAG 通过为 LLM 提供从相关数据中检索到的额外上下文来工作，从而无需对模型进行微调，就可以生成更为准确的回答。

这里展示了一个简单的 RAG 示意图：

  • 外部数据被转换为嵌入向量，存储在数据库中。更新这些嵌入向量比微调整个模型要简单、快捷和经济。

  • 此外，不需要微调的优势还在于，当有更强大的 LLM 出现时，你可以随时更换模型，或者如果需要更快的推理速度，可以选择使用一个更小的版本。

下面，我们将使用开源的 LLM、嵌入模型和 LangChain 来展示如何构建一个 RAG。

准备数据

这个示例中，我们将加载 PEFT 开发库 中的所有问题（无论是开放还是已关闭的）。

首先，你需要获得一个 GitHub 个人访问令牌 来访问 GitHub API。

from getpass import getpass
ACCESS_TOKEN = getpass("YOUR_GITHUB_PERSONAL_TOKEN")

然后，我们会加载 huggingface/peft 中的所有问题：

  • 默认情况下，

拉取请求也会被当作问题，这里我们选择通过设置 include_prs=False 来排除它们。

  • 设置 state = "all" 意味着我们将获取所有的开放和关闭的问题。
from langchain.document_loaders import GitHubIssuesLoader

loader = GitHubIssuesLoader(repo="huggingface/peft", access_token=ACCESS_TOKEN, include_prs=False, state="all")

docs = loader.load()

GitHub 问题的内容可能比嵌入模型能处理的输入要长，我们需要将这些内容适当分割。

文档分割的常见方法是定义固定大小的块，并决定这些块之间是否应该有重叠，以保持语义连贯性。我们将使用推荐的 RecursiveCharacterTextSplitter 来完成这一任务。

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=30)

chunked_docs = splitter.split_documents(docs)

创建嵌入和检索器

现在文档已经被切分为合适的大小，我们可以建立一个包含这些文档嵌入的数据库。

我们将使用 HuggingFaceEmbeddings 和 BAAI/bge-base-en-v1.5 来创建文档块的嵌入。还可以在 Hub 上找到许多其他的嵌入模型，可以通过查看 Massive Text Embedding Benchmark (MTEB) 排行榜 来选择性能最优的模型。

创建向量数据库时，我们使用的是 FAISS，这是 Facebook AI 开发的一个库，专门用于进行密集向量的相似性搜索和聚类，非常适合我们的需求。FAISS 是目前处理大规模数据集中最常使用的 NN 搜索库之一。

from langchain.vectorstores import FAISS
from langchain.embeddings import HuggingFaceEmbeddings

db = FAISS.from_documents(chunked_docs, HuggingFaceEmbeddings(model_name="BAAI/bge-base-en-v1.5"))

我们需要一种机制来根据非结构化的查询返回文档。因此，我们将利用 db 作为后端来实现一个检索器：

  • 设置 search_type="similarity" 表明我们希望在查询和文档之间执行相似性搜索。
  • search_kwargs={'k': 4} 的设置让检索器返回最相关的前四个结果。
retriever = db.as_retriever(search_type="similarity", search_kwargs={"k": 4})

现在，向量数据库和检索器已经准备就绪，我们接下来设置模型部分。

加载量化模型

我们选择了 HuggingFaceH4/zephyr-7b-beta，一个小型但强大的模型。

随着每周发布的新模型数量的增加，你可能会想用最新最好的模型来替换现有的。跟踪最新的开源 LLM 最好的方式是查看 Open-source LLM 排行榜。

为了加快推理速度，我们将使用该模型的量化版本。

import torch
from transformers import AutoTokenizer, AutoModelForCausalLM, BitsAndBytesConfig

model_name = "HuggingFaceH4/zephyr-7b-beta"

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True, bnb_4bit_use_double_quant=True, bnb_4bit_quant_type="nf4", bnb_4bit_compute_dtype=torch.bfloat16
)

model = AutoModelForCausalLM.from_pretrained(model_name, quantization_config=bnb_config)
tokenizer = AutoTokenizer.from_pretrained(model_name)

设置 LLM 链

现在我们已经拥有了设置 LLM 链所需的所有部件。

首先，我们使用已加载的模型及其 tokenizer 来创建一个文本生成管道。

然后，我们创建一个符合模型格式的提示模板，以确保在替换模型时，也能使用适当的格式。

from langchain.llms import HuggingFacePipeline
from langchain.prompts import PromptTemplate
from transformers import pipeline
from langchain_core.output_parsers import StrOutputParser

text_generation_pipeline = pipeline(
    model=model,
    tokenizer=tokenizer,
    task="text-generation",
    temperature=0.2,
    do_sample=True,
    repetition_penalty=1.1,
    return_full_text=True,
    max_new_tokens=400,
)

llm = HuggingFacePipeline(pipeline=text_generation_pipeline)

prompt_template = """
<|system|>
Answer the question based on your knowledge. Use the following context to help:

{context}

</s>
<|user|>
{question}
</s>
<|assistant|>

 """

prompt = PromptTemplate(
    input_variables=["context", "question"],
    template=prompt_template,
)

llm_chain = prompt | llm | StrOutputParser()

还可以使用 tokenizer.apply_chat_template 功能，将一系列消息（格式为字典：{'role': 'user', 'content': '(...)'}）转换为具有适当聊天格式的字符串。

最后，我们需要将 llm_chain 与检索器结合，创建一个 RAG 链。我们将原始问题以及检索到的上下文文档一起传递到最终的生成步骤中：

from langchain_core.runnables import RunnablePassthrough

retriever = db.as_retriever()

rag_chain = {"context": retriever, "question": RunnablePassthrough()} | llm_chain

结果对比

让我们看看 RAG 在回答特定图书馆问题时能带来什么样的改变。

question = "How do you combine multiple adapters?"

首先，我们看看仅使用模型本身，未添加任何上下文时，我们能得到什么答案：

llm_chain.invoke({"context": "", "question": question})

可以看到，模型将问题解读为与物理计算机适配器相关的问题。
然而在 PEFT 的背景下，“适配器”实际上指的是 LoRA 适配器。接下来，让我们看看添加来自 GitHub 问题的上下文后，模型是否能提供更相关的答案：

rag_chain.invoke(question)

正如我们所见，相同的模型在加入了上下文后，确实能够提供更准确、更有针对性的回答。

这一点尤为重要，结合多个适配器用于推理的功能已经被加入到库中，这一信息可以在文档中找到。因此，在 RAG 的下一次迭代中，包含文档嵌入可能会是一个值得考虑的选择。
